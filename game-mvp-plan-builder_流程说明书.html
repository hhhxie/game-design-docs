一、技能概述

1.1 用途

本技能用于生成结构化 Plan 提示词，可直接粘贴到 VS Code CodeBuddy 插件中，用于自动生成完整的网页端游戏 MVP 原型（单 HTML 文件）。

1.2 触发方式

指令触发：@skill:game-mvp-plan-builder

关键词触发：

  • 生成游戏MVP提示词
  • game MVP plan
  • 帮我生成游戏原型提示词
  • MVP plan prompt
  • 为游戏原型生成plan

1.3 核心能力

  • 支持有完整思路和无完整思路两种工作模式
  • 自动从用户描述中提取游戏设计参数
  • 逐维度确认/修改，提供多选项而非开放式问答
  • 生成标准化 Plan 提示词（含技术要求、文件结构、实现步骤）

二、完整流程图

【入口判断】用户触发技能 → Step 0 前置判断（是否有完整思路/文档？）
【分支 A — 有完整思路】
接收用户描述 → 信息提取（6 个维度）→ 逐维度确认（A/B/C/自定义）→ 汇总 → 生成 Plan
【分支 B — 无完整思路】
Step 1/6 游戏类型 → Step 2/6 游戏视角 → Step 3/6 2D/3D → Step 4/6 场景/世界观 → Step 5/6 核心机制 → Step 6/6 操作输入与反馈 → 生成 Plan
【最终输出】
生成 Plan 提示词 → 用户确认/修改 → 输出最终提示词（复制粘贴至 VS Code CodeBuddy）

三、详细流程说明

Step 0：前置判断

目的：判断用户是否有完整的游戏设计思路，选择对应工作分支。

执行动作：

  1. 展示欢迎语
  2. 询问：是否有完整思路/信息需要输入？
  3. 提供选项：
    • [A] 有 → 进入信息提取模式
    • [B] 没有 → 进入六步问答模式

分支 A：信息提取模式

A-1：接收并分析用户输入

输入形式：

  • 大段文字描述（自由文本）
  • 结构化策划文档（Markdown / 纯文本）
  • 关键词组合（尽力提取）

提取维度（共 6 个）：

#维度提取目标无法提取时的处理
1游戏类型Platformer / Shooter / Racing / Fighting / Puzzle / ARPG 等记 [未识别]，后续转为问答
2游戏视角Top-down / Side-scrolling / FPS / Third-person / Isometric 等记 [未识别]
32D/3D2D (Canvas) / 3D (Three.js) / 2.5D记 [未识别]
4场景/世界观世界观/场景风格 + 场景核心物件清单 + 场景范围/边界记 [未识别]，后续转为问答
5核心机制胜负条件 + 玩法循环 + 特色机制部分提取，缺失部分标记
6输入/反馈摄像头体感 / 手柄陀螺仪 / 键盘+鼠标 / 手柄（4选1）记 [未识别]，后续转为问答

A-2：逐维度确认（核心创新点）

每个维度独立确认，格式如下：

【维度确认 X/6】{维度名称}

📋 从你的描述中，我提取到：
{提取结果 或 "[未识别，请提供]}

请从以下选项中选择，或输入自定义内容：
- [A] {推荐选项1，与提取结果匹配}
- [B] {推荐选项2，同类别另一常见选择}
- [C] {推荐选项3，不同风格的选择}
- [D] 自定义（请直接描述）

各维度推荐选项参考表：

  • ① 游戏类型：A 动作 RPG（ARPG）/ B 平台跳跃（Platformer）/ C 射击游戏（Shooter）/ D 自定义
  • ② 游戏视角：A 俯视角（Top-down）/ B 侧视角（Side-scrolling）/ C 第三人称跟随 / D 自定义
  • ③ 2D / 3D：A 2D（Canvas 2D API）/ B 3D（Three.js）/ C 伪3D（2.5D，等距视角）/ D 自定义
  • ④ 场景/世界观：A（与提取结果一致）/ B 霓虹都市（科技风）/ C 热带雨林（自然风）/ D 自定义
  • ⑤ 核心机制：A（按提取结果填写）/ B 简化版（仅胜负条件+基础玩法循环）/ C 复杂版（加入装备/技能/升级系统）/ D 自定义
  • ⑥ 操作输入与反馈：A 📷 摄像头体感 / B 🎮 手柄陀螺仪 / C ⌨️🖱️ 键盘+鼠标 / D 🎮 手柄 / E 自定义

分支 B：六步问答模式

Step 1/6：游戏类型

【Step 1/6】游戏类型是什么？
常见类型参考（可自定义）：
- 🏃 平台跳跃（Platformer）
- 🎯 射击（Shooter / Bullet Hell）
...
- 其他（请直接描述）

Step 4/6：场景与世界观（含物件与范围）

  • ① 世界观 / 场景风格 / 背景：例如霓虹都市（科技感）、热带雨林（自然冒险）等
  • ② 场景中的物件：场景中应该有哪些关键物件/道具/元素
  • ③ 场景范围 / 边界：游戏场景有多大？边界如何定义？

Step 6/6：操作输入与反馈

  • 输入方式（4 选 1）：📷 摄像头体感 / 🎮 手柄陀螺仪 / ⌨️🖱️ 键盘+鼠标 / 🎮 手柄
  • 反馈方式：由技能根据已设计的游戏类型动态生成推荐选项

四、提示词生成规范

4.1 输出结构（模板）

生成的 Plan 提示词必须包含以下章节：

# 游戏 MVP 原型开发 Plan

## 目标
（一句话说明目标）

## 游戏基础信息
- 游戏类型：{填入}
- 视角：{填入}
- 渲染方式：{填入}
- 场景/世界观：{填入}
- 场景物件：{填入}
- 场景范围：{填入}

## 核心机制
- 目标/胜负条件：{填入}
- 核心玩法循环：{填入}
- 特色机制：{填入 或 暂无}

## 操作输入
- 输入方式：{填入}
- 反馈方式：{填入}

## 技术要求
- 单 HTML 文件（内联 CSS/JS）
- {按 2D/3D/体感 选择对应技术条款}
- 游戏分辨率：1280×720（响应式）
- 包含：主循环、碰撞检测、UI、状态机

## 文件结构
输出单文件 {name}.html

## 实现步骤（Plan 分解）
（6-12 步，根据复杂度调整）

## 额外要求
- 详细中文注释
- 常量提取到顶部 CONFIG 对象
- 中文命名（游戏名/角色名/技能名）
- 基础视觉美化

4.2 技术条款选择规则

渲染方式保留条款删除条款
2D使用原生 Canvas 2D APIThree.js 条款、MediaPipe 条款
3D使用 Three.js（CDN引用）Canvas 2D 条款
体感使用 MediaPipe Pose（CDN引用）—
需要音效追加 使用 Howler.js（CDN引用）—

4.3 实现步骤数量规范

游戏复杂度推荐步骤数适用类型
简单（单机制）6-8 步平台跳跃、Flappy Bird 类
中等（多系统）8-10 步射击、赛车、益智解谜
复杂（多系统+AI）10-12 步ARPG、格斗、策略类

五、异常处理规则

5.1 信息提取失败

情况处理方式
某维度完全无法提取自动转为该维度的分支 B 问答格式（而非阻塞流程）
用户描述与提取结果不符以用户最新输入为准，更新该维度
用户输入含糊给出默认推荐值并继续

5.2 分支切换

  • 分支 A 中若超过 3 个维度无法提取 → 询问用户是否切换到分支 B
  • 分支 B 进行中，用户突然粘贴大段描述 → 自动切换回分支 A，重新提取

5.3 技术依赖提醒

触发条件提醒内容
用户选择 3D"注意：Three.js 需从 CDN 加载，首次运行需要网络连接"
用户选择体感（MediaPipe）"注意：MediaPipe Pose 需要摄像头权限，且对设备性能要求较高"
用户选择音效"注意：Howler.js 需从 CDN 加载，若需离线运行可提前下载至本地"

九、注意事项

  • 每步问答结束后立即进入下一步，不要等待用户催促
  • 提取结果与用户纠正冲突时，以用户最新输入为准
  • Plan 提示词的实现步骤数量需根据游戏复杂度灵活调整
  • 中文命名：游戏名称、角色名称、技能名称统一使用中文
  • 常量提取：所有可调参数必须放入顶部 CONFIG 对象
  • Step 6 输入方式：固定为 4 选 1，不再使用原来的多选模式
  • Step 6 反馈方式：必须根据用户已选择的游戏类型动态选取推荐选项
  • Step 4 场景物件询问：场景物件和范围是生成场景资产清单的关键依据，不可跳过
