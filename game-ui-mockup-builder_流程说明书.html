一、技能概述

1.1 用途

本技能通过读取游戏流程图或策划大纲，构建可交互的 HTML 游戏 UI 原型（单文件），在一个 HTML 中模拟所有界面及界面间的导航逻辑，无需实际游戏玩法。

1.2 触发方式

指令触发：@skill:game-ui-mockup-builder

关键词触发：

  • 制作UI原型
  • 游戏界面模拟
  • UI流程图
  • 构建界面原型
  • game UI mockup

1.3 核心能力

  • 支持多种输入：流程图图片 / .md/.docx 策划大纲 / 文字描述 / 无输入
  • 逐个界面设计方案确认（每个界面提供 2 个方案 + 自定义）
  • 复用参考实现的设计模式（CSS 变量 / 响应式缩放 / 键盘导航）
  • 结果模拟：胜利/失败/平局等不同操作结果的模拟展示
  • 完整导航状态机：界面切换动画 + 键盘/触摸操作

二、完整流程概览

【Phase 0 — 输入解析】
判断输入类型（流程图图片/md文档/文字描述/无输入）→ 提取界面清单 + 跳转条件
【Phase 1 — 逐个界面设计方案确认】
列出所有界面 → 每个界面提供 2 个方案供选择 → 用户确认或自定义 → 汇总确认表
【Phase 2 — UI 设计细化】
调用 ui-ux-pro-max skill → 确定布局/色彩/字体/交互状态 → 复用参考文件 CSS 模式
【Phase 3 — 导航逻辑实现】
实现 JavaScript 状态机 → showView() 切换界面 → 支持键盘/触摸/超时跳转
【Phase 4 — 结果模拟】
定义操作→结果路径 → 模拟胜利/失败/平局/超时等 → 弹窗覆盖层展示
【Phase 5 — 生成 HTML 文件】
集成所有界面 + 导航 + 模拟逻辑 → 响应式缩放 → 键盘导航 → 无实际游戏代码

三、详细流程说明

Phase 0 — 输入解析

判断输入类型：

输入类型处理方式
流程图图片描述图片内容，提取所有节点和连接关系
.md/.docx 策划大纲读取"界面流程"或"UX 流程"章节
直接文字描述按用户提供的大纲解析
无输入询问用户描述游戏的主要界面流程

提取内容：

  • 所有界面名称（如：主菜单 → 球队选择 → 比赛 → 结算）
  • 每个界面的核心元素（按钮、列表、表单等）
  • 界面间的跳转条件（点击什么 → 去哪里）

Phase 1 — 逐个界面设计方案确认

核心流程：每个界面逐一设计确认，生成 2 个方案供用户选择，支持自定义。

执行步骤：

  1. 列出所有界面，确认数量和顺序 
    === 界面清单确认 ===
| 编号 | 界面名称 | 进入条件 | 离开条件 |
|------|---------|---------|---------|
| 1 | 主菜单 | 游戏启动 | 点击开始 → 进入球队选择 |
| 2 | 球队选择 | 从主菜单 | 选择完成 → 进入比赛 |
| 3 | 比赛界面 | 从球队选择 | 比赛结束 → 进入结算 |
| 4 | 结算界面 | 从比赛界面 | 点击返回 → 回到主菜单 |
  2. 逐个界面设计，对每个界面生成 2 个方案 
    === 界面 1/4：主菜单 ===

【方案 A】
布局：垂直居中布局，大标题在上，3个按钮纵向排列
操作：鼠标点击 / 键盘方向键+回车
特色：背景动态粒子动画，按钮hover发光效果

【方案 B】
布局：全屏海报风格，底部固定按钮栏
操作：触摸滑动 / 鼠标点击
特色：背景轮播图，按钮底部固定不动

请选择界面「主菜单」的布局方案：
[ A ] 方案A：垂直居中布局，键盘+鼠标操作
[ B ] 方案B：海报风格，触摸+点击操作
[ C ] 自定义：请描述你的想法
  3. 循环执行，直到所有界面确认完毕
  4. 输出最终汇总确认表

Phase 2 — UI 设计细化

基于 Phase 1 确认的方案，使用 UI/UX Pro Max 原则进行细化：

  1. 调用 ui-ux-pro-max skill 获取设计建议
  2. 确定每个界面的：
    • 布局结构（CSS Grid / Flexbox）
    • 色彩方案（继承参考文件的 CSS 变量模式）
    • 字体层级
    • 交互状态（hover / active / disabled / selected）
  3. 复用参考文件的通用设计模式：
    • CSS 变量集中管理（:root）
    • 深色游戏风配色（深蓝/黑色底色 + 金色高亮）
    • 卡片式布局 + 毛玻璃效果（backdrop-filter: blur）
    • 响应式缩放（1920×1080 基准）

参考配色方案：

:root {
  --bg: #050b16;       /* 主背景深蓝黑 */
  --panel: #0d1d33;    /* 面板背景 */
  --panel2: #132944;   /* 二级面板 */
  --text: #f5fbff;     /* 主文字 */
  --muted: #8da4bd;    /* 次要文字 */
  --gold: #ffd166;     /* 高亮/选中 */
  --green: #39d98a;   /* 成功/确认 */
  --red: #ff5c5c;      /* 失败/警告 */
  --blue: #49a0ff;     /* 信息/1P */
  --atk: #ff6b6b;     /* 攻击 - 红 */
  --def: #4ecdc4;     /* 防守 - 青 */
  --spd: #ffe66d;     /* 速度 - 黄 */
}

Phase 3 — 导航逻辑实现

JavaScript 状态机模板：

let currentView = 'mainMenu';
const VIEWS = {
  mainMenu:   { el: '#mainMenu',   next: 'teamSelect' },
  teamSelect: { el: '#teamSelect',  next: 'match' },
  match:      { el: '#match',       next: 'result' },
  result:     { el: '#result',      next: 'mainMenu' },
};

function showView(viewName) {
  Object.values(VIEWS).forEach(v => $(v.el).style.display = 'none');
  $(VIEWS[viewName].el).style.display = 'grid';
  currentView = viewName;
}

导航触发方式（根据 Phase 1 用户选择的操作方式）：

触发方式实现方式
按钮点击onclick="showView('targetView')"
键盘按键keydown 事件（方向键 + 回车）
触摸滑动touch 事件
超时自动跳转setTimeout(() => showView('next'), 3000)

Phase 4 — 结果模拟

操作可能结果模拟方式
开始比赛胜利/失败/平局随机生成，可设置玩家保胜概率
选择球队确认/取消高亮选中 + 确认按钮激活
抽奖/开包稀有/普通/重复随机结果 + 动画展示
购买道具成功/金币不足检查虚拟余额

结果展示模式：

  • 弹窗覆盖层（.overlay.show）
  • 结果卡片（.result-card）
  • 确定按钮关闭弹窗并返回

通用结果生成函数：

function simulateResult(options = {}) {
  const { playerWinRate = 0.6, drawRate = 0.2, maxScore = 5 } = options;
  const rand = Math.random();
  if (rand < drawRate) {
    // 平局
    playerScore = opponentScore = Math.floor(Math.random() * 4);
  } else if (rand < drawRate + playerWinRate * (1 - drawRate)) {
    // 玩家胜
    playerScore = Math.floor(Math.random() * 3) + 1;
    opponentScore = Math.floor(Math.random() * playerScore);
  } else {
    // 对手胜
    opponentScore = Math.floor(Math.random() * 3) + 1;
    playerScore = Math.floor(Math.random() * opponentScore);
  }
  return { playerScore, opponentScore };
}

Phase 5 — 生成 HTML 文件

必须包含的功能：

  • 响应式缩放（resizeStage() 函数）
  • 界面切换动画（CSS transition / animation）
  • 键盘导航支持（方向键 + 回车）
  • 结果弹窗系统（modal overlay）

四、输出检查清单

生成 HTML 后，自动验证：

  • 所有流程图中的界面都已实现
  • 每个界面都可以通过某种操作进入
  • 导航逻辑符合流程图描述
  • 结果模拟覆盖主要分支（胜/负/平）
  • 键盘导航可用（如有需要）
  • 响应式缩放正常（窗口大小变化）
  • 无实际游戏玩法代码（仅 UI 模拟）

五、异常处理规则

情况处理方式
用户选择 [C] 自定义详细询问其想法并记录，更新该界面的自定义描述
用户提供文字描述而非图片使用流程图描述模板解析
无输入主动询问用户描述游戏的主要界面流程
界面数量过多（>10）分批确认，每批不超过 5 个界面

八、注意事项

  • Phase 1 逐界面确认是核心：每个界面必须独立确认，不能跳过
  • 结果模拟不等于游戏玩法：仅模拟操作结果，无真实游戏逻辑
  • 参考文件是 CSS/JS 的直接来源：配色、布局、动画模式均从参考文件提取
  • 键盘导航是加分项：如 Phase 1 用户选择了键盘操作，必须实现方向键+回车导航
  • 响应式缩放必须实现：使用参考文件的 scale = min(vw/1920, vh/1080) 模式
  • Phase 5 完成后输出检查清单：逐项验证，确保所有界面可导航
