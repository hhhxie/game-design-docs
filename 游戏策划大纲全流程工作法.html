游戏策划大纲
全流程工作法

根据游戏情况选择合适的路径。有成熟玩法走路径 A（直接生成大纲），纯创意构想走路径 B（先 MVP 验证）。两条路径最终都产出可验证、可迭代、可分享的游戏设计资产。

Purpose

把模糊创意变成可交付开发的策划资产

传统文档容易停留在文字层面，而这套流程要求每个关键设计都经过原型验证、网页化表达和在线部署。

01

沟通成本高

策划用文字描述玩法，开发再理解实现，双方容易出现认知偏差。

02

细节难验证

交互体验是否流畅，常常要等代码完成后才暴露问题。

03

UI 设计滞后

流程和界面在开发后期才明确，修改成本极高。

04

文档实现脱节

文档写完后没有持续校准，最终与实际体验不一致。

Skills

三大技能分工：文档生成、玩法验证、UI 模拟

三个技能不是孤立使用，而是组成一条从文本到原型再到交付的流水线。

game-design-doc-generator

策划大纲生成器

通过结构化问答生成符合行业规范的游戏策划大纲。

  • 支持动作、体育、养成、休闲、RPG 等类型
  • 遵循 10 节标准文档结构
  • 每个问题提供 1-3 个参考选项
  • 自动检查遗漏章节和模糊表述
输出：.md 策划大纲
game-mvp-plan-builder

MVP 计划构建器

收集核心机制信息，产出用于生成游戏原型的 PLAN 提示词。

  • 六步引导式问答收集核心机制
  • 三要素场景设计（风格+物件+范围）
  • 生成可交给 CodeBuddy 的 PLAN 提示词
  • 严格的 P0/P1/P2 MVP 范围管控
输出：PLAN 提示词
game-ui-mockup-builder

游戏 UI 流程模拟器

生成可交互的 UI 流程原型，模拟所有界面跳转和结果分支。

  • 解析界面清单、核心元素、跳转条件
  • 逐个界面提供 A/B 设计方案确认
  • 支持键盘、鼠标、触摸操作
  • 模拟胜利、失败、平局、超时等结果
输出：.html UI 原型
Two Paths

两条路径，先选再走

在开始之前，先判断你的游戏属于哪种情况。不同的起点，决定了不同的流程路径。

✅ 推荐路径

情况一：有成熟玩法 + 有参照竞品

当你已经有明确的核心玩法，并且能找到对应的体感操作参照竞品时，可以直接生成策划大纲。

  • 核心玩法已经过市场验证（如：体感足球、体感拳击）
  • 已有类似游戏可以参考操作方式（如：Just Dance）
  • MVP 原型是可选项，仅当需要验证技术创新时才生成
流程：7~8 步 · 更短更快
🔧 验证路径

情况二：纯创意构想 / 未改编类型

当你的游戏是全新创意构想，或需要将传统游戏改编为体感操作时，需要先通过 MVP 验证。

  • 玩法是全新的创意，没有参照竞品
  • 需要探索全新的动作映射方案
  • MVP 验证是必要环节，确保操作性可行
流程：10 步 · 先验后策

Path A

路径 A：直接大纲路径（情况一）

有成熟玩法和参照竞品时，直接生成策划大纲，MVP 可选。

1

生成策划大纲

使用 game-design-doc-generator，通过问答形成规范化游戏策划文档。

产出

.md 策划大纲，统一语言、减少歧义。

2（可选）

使用 game-mvp-plan-builder

收集核心机制信息，产出 PLAN 提示词，然后交给 CodeBuddy 生成 MVP 原型。

价值

当需要验证技术创新时使用，可选步骤。

3

VS Code + CodeBuddy 迭代

边体验边调整数值、UI、交互和遗漏功能，直到满足需求。

效果

快速反馈、即时修改、低成本验证。

4

生成核心机制文档网页

基于调整后的 MVP，把机制、数值、规则转化为可视化 HTML 文档。

产出

带图表、流程、属性条的网页文档。

5

生成 UI 流程模拟原型

使用 game-ui-mockup-builder 生成完整界面流程和结果模拟。

价值

快速生成 UI 示意图，同时说明操作流程。

6

调整 UI 原型

继续用 VS Code + CodeBuddy 精修布局、跳转逻辑和交互细节。

效果

形成可作为 UI/UX 参考标准的最终原型。

7

部署至 GitHub Pages

将所有网页文件发布为可访问 URL，方便评审、演示和远程协作。

产出

在线文档与原型矩阵。

8

整合链接至大纲文档

把 MVP、UI 原型、核心机制网页等链接汇总到同一份交付清单。

产出

一站式索引文档，便于交付和查阅。

Path B

路径 B：MVP 验证路径（情况二）

纯创意构想或未改编类型，必须先通过 MVP 原型验证操作可行性，再生成正式策划文档。

1

使用 game-mvp-plan-builder

收集核心机制信息，产出可用于生成 MVP 原型的 PLAN 提示词。

产出

结构化 PLAN 提示词。

2

VS Code CodeBuddy 生成 MVP

将 PLAN 提示词丢给 VS Code 中的 CodeBuddy 插件，生成游戏原型。

产出

可运行的 .html MVP 原型。

3

体验验证 + 迭代修改

在 VS Code 中打开文件，直接对话修改，直到核心玩法和操作体验满足需求。

关键

✅ 验证通过后才进入下一步。

4

从 MVP 提取核心玩法说明

从验证后的 MVP 文件中提取：玩法机制、数值公式、操作逻辑。

产出

经过验证的核心机制清单。

5

生成策划大纲

使用 game-design-doc-generator，基于验证过的核心玩法生成正式策划文档。

产出

.md 策划大纲。

6

生成核心机制文档网页

把机制、数值、规则转化为可视化 HTML 文档。

产出

带图表的网页文档。

7

生成 UI 流程模拟原型

使用 game-ui-mockup-builder 生成完整界面流程。

价值

同时生成 UI 示意图和操作流程。

8

调整 UI 原型

VS Code + CodeBuddy 精修布局和交互细节。

效果

满足所有 UI/UX 需求。

9

部署至 GitHub Pages

发布为可访问 URL，方便评审和演示。

产出

在线原型矩阵。

10

整合链接至大纲文档

汇总所有链接到交付清单。

产出

一站式索引文档。

Value

每一步为什么要这么做

流程的重点不是多产出几个文件，而是让设计问题尽早暴露、尽早验证、尽早统一。

纲

策划大纲是项目蓝图

把模糊创意转成清晰设计语言，例如把"体感足球"拆解为摆臂加速、挥臂踢球、身体转向等可实现动作。

验

MVP 验证是核心环节（路径B）

文字描述听起来合理，但举手踢球是否自然、摆臂灵敏度是否合适，必须通过真实操作判断。低成本试错，发现问题就推翻重来。

改

VS Code + CodeBuddy 加速迭代

在 HTML 文件中直接修改代码，浏览器刷新即可查看效果，大幅降低调整成本。

图

网页文档提升理解效率

用图表、属性条、流程图展示规则和数值，比纯文字更适合团队评审和远程沟通。

UI

UI 模拟避免后期返工

在开发前验证界面层级、按钮跳转和结果分支，避免做到一半才发现流程不符合需求。

链

在线链接形成交付矩阵

把所有网页部署到 GitHub Pages 后，团队可直接打开链接体验、评审、演示和同步意见。

Two Paths

两条路径，一个目标

路径A直接大纲：适合有成熟玩法和参照竞品的项目，快速产出策划文档。
路径BMVP验证：适合纯创意构想或未改编类型，先验证再策划。
最终产出策划大纲 + 核心机制文档 + UI原型 + 部署链接。
交付目标可体验、可迭代、可分享的游戏设计资产。
闭环迭代每一步产出都是下一步输入，形成完整设计流水线。
Summary

两条路径对比

A

路径 A：直接大纲路径

  • 适用：有成熟玩法 + 有参照竞品
  • MVP 原型：可选
  • 流程：7~8 步
  • 优势：快速
B

路径 B：MVP 验证路径

  • 适用：纯创意构想 / 未改编类型
  • MVP 原型：必要

  • 优势：可验证
